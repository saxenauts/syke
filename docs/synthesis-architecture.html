concept

Synthesis is the intelligence layer. It sits on top of the observe layer (the immutable evidence) and builds memory — stories, connections, and a routing map — from raw events. The agent has full bash access to the timeline, 2 MCP tools for writing memory and committing cycles, and a skill file that serves as its control plane.

The Three Spaces

Syke operates across three data spaces with different mutability rules.

Space 1: Timeline

Immutable. Read-only.

The events ledger from the observe layer. Append-only. The agent queries it via bash but never modifies it. Self-observation traces stored as cycle_annotations.

199,138 events · 25+ columns

11 sources · 31 models

5,488 sessions · FTS5 index

Space 2: Graph Space

Mutable. Agent-shaped.

Stories as nodes, connections as edges. The agent creates, updates, supersedes, deactivates memories and links them together via the memory_write dispatch tool.

717 memories · 382 links

Markdown free-form text

FTS5 index (trigger-synced)

Space 3: Router (Memex)

Learned. Dual-readable.

A routing map rewritten each synthesis cycle. Serves humans as a dashboard and agents as a navigation table. Emerged through information pressure.

1 per user · Rewritten each cycle

Control Plane Design

The fixed control plane is separated from the learned layer. This enables systematic experimentation and measurement.

Current State

The synthesis skill is extracted to syke/memory/skills/synthesis.md (93 lines, variant: map). It describes affordances in values language — what to notice, how to think — not tool commands.

SHA256 hash of the skill file is stored on each cycle_record for provenance. The runtime envelope injects the skill content along with backlog stats, DB path, and temporal context.

Key principle: "Not instructions, primitives. This is the truth. This is what we are measuring."

Why This Separation Matters

Measurability: If the control plane changes every cycle, you can't measure what's working. Fixed primitives let you isolate what the agent is learning.

Ablation: Variant overlays let you test different framings of the same primitives. "map" skill vs "static" skill — same core, different language.

Evolution: GEPA-style prompt optimization requires versioning. SHA256 hash per cycle, stored with results, enables evolutionary search over the control plane.

Completion Contract

The agent must call commit_cycle exactly once. This is the only path to persist the memex and advance the synthesis cursor.

Post-run validation: After the agent run completes, the harness checks whether commit_cycle was called (detected via both the tool handler nonlocal and stream inspection). If it was: process the result. If not: return status="incomplete" — no error raised, the cycle is simply marked incomplete in the cycle_record.

commit_cycle semantics:

status='completed' + non-empty content → updates memex, advances cursor
status='completed' without content → advances cursor, memex unchanged
status='failed' → cycle recorded as failed, no memex update
Agent never calls commit_cycle → post-run detection marks cycle as incomplete

commit_cycle — finalize synthesis

Daemon Cycle

The daemon runs every 15 minutes (configurable). It syncs all sources, triggers synthesis if enough new events exist, and distributes the updated memex. Implemented in sync.py (205 lines).

Every 15 minutes (DAEMON_INTERVAL = 900s): +- 1. Health ------ Check all harness connections | +- 2. Heal -------- Repair broken connections | +- 3. Reconcile --- Sync all sources via observe adapters | (each source: detect -> discover -> ingest -> dedup) | +- 4. Synthesize -- If >=5 new events (SYNC_EVENT_THRESHOLD): | -> synthesize(db, user_id) | -> agent runs, writes memories, commits cycle | +- 5. Distribute -- Push updated memex to all harnesses: -> CLAUDE.md context file -> Skills directories -> Hermes, Claude Desktop adapters

Cursor & Backlog Management

The synthesis cursor tracks progress through the timeline. It determines what's "new" for each cycle.

synthesis_cursor table: user_id (PK) | last_event_id | updated_at Before synthesis: count = events WHERE id > last_event_id <- backlog size newest = MAX(id) FROM events <- will become new cursor Agent receives: "Pending events since last synthesis: {count}" "Cursor (last processed event ID): {cursor_id}" "Query events with: id > '{cursor_id}'" After synthesis (commit_cycle status='completed'): cursor advances to newest event available at cycle start Deterministic - based on what was AVAILABLE, not what was READ Agent decides what to query within the window If commit_cycle not called: cursor does NOT advance - events reprocessed next cycle

Distribution

After synthesis, the memex is distributed to all connected harnesses so every agent has the latest context.
